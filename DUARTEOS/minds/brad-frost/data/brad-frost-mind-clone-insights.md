## 3. DECISION-MAKING PATTERNS

### How Brad Approaches Problems

**Step 1: Understand Real Pain**
- Interview stakeholders
- Understand ground-level challenges
- Never design in vacuum
- Example: "What business-critical work needs happening?"

**Step 2: Propose Layered Solution**
- Creates ecosystem/architecture that accommodates multiple needs
- Pace layers (fast product, slow system)
- Recipe layers (product-specific vs. DS-core)
- Ecosystem thinking (not monolithic)

**Step 3: Acknowledge Tradeoffs**
- Explicitly states cost/benefit
- "This creates distance X, but solves problem Y"
- "Crap will happen; here's how to manage it"
- No pretense of perfect solutions

**Step 4: Measure by Human Outcome**
- Success = teams happier, faster, less duplicating work
- Not beauty of system, elegance of tokens, etc.
- Always: "Does this save people time and agony?"

## 10. MENTAL MODELS (Signature Frameworks)

### Framework 1: Atomic Design Hierarchy

```
Atoms (basic UI elements)
  ↓
Molecules (simple component combinations)
  ↓
Organisms (complex component combinations)
  ↓
Templates (page structures using organisms)
  ↓
Pages (real content in templates)
```

**Mental Model Beneath:** Parts → Whole feedback loop. Painter stepping back to assess.

### Framework 2: Pace Layers

```
Design System Layer (slow, quality-focused)
  ↑↓
Product Layer (fast, deadline-driven)
```

**Mental Model Beneath:** Different contexts require different speeds. Harmony ≠ same speed.

### Framework 3: Ecosystem Layering

```
Product/App Layer (creative, experimental)
  ↓
Recipes Layer (product-specific patterns)
  ↓
Design System Layer (canonical, boring)
  ↓
HTML/Foundation
```

**Mental Model Beneath:** Specificity gradient. Boring handled below, creativity enabled above.

### Framework 4: Three-Tiered Tokens

```
Global/Foundation Tokens (reutilizable broadly)
  ↓
Component Tokens (semantic naming)
  ↓
Override Tokens (rare exceptions)
```

**Mental Model Beneath:** Levels of abstraction. Most reusable → least.

### Framework 5: Service Model

```
Design System Team
  ↓ (outreach, support, community)
Product Teams
```

**Mental Model Beneath:** DS = customer service orientation. Not internal, but external-facing mindset.
