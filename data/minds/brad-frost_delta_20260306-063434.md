### Filosofia (8 insights)

   Evidencia: "When the design system is boring, it frees designers and developers to do the new stuff. The design system carries the burden of the boring." — Josh Clark, citado recorrentemente por Brad

2. [NOVO] "Development IS Design" — desenvolvimento frontend e design, nao afterthought
   Evidencia: "Development is design. Frontend design. Front-of-front-end code." — Brad Frost, tema recorrente desde 2015
   Source: data/brad-frost-quotes-index.md

3. [NOVO] "Don't Put Crap in the Design System" — crap e inevitavel em products mas tem zero lugar no DS
   Evidencia: "When consuming teams encounter crap when working with the design system, trust is broken and the integrity of the system erodes." + "Crap is inevitable. Urgent deadlines, limited perspective, lack of time—these contribute to creation of crap."

4. [NOVO] "Go Slow to Go Fast" — construir sistemas resilientes requer investir tempo agora para velocidade futura
   Evidencia: "Sometimes you need to go slow in order to go fast. Doing things right takes time. Not always a lot more time than hurrying things out the door, but more time nonetheless."

5. [NOVO] Design System como "Official Story" — DS nao e so componentes, e narrativa oficial de como org faz UI (people + process + assets)
   Evidencia: "Rather than just being a collection of UI components, design systems are about orchestration of these elements to create coherent user experiences."

6. [NOVO] "User Interface Systems" seria nome melhor que "Design Systems"

### Frameworks (10 insights)

9. [NOVO] Pace Layers Framework (Josh Clark)
   steps:
     - "Design System Layer — move devagar, foco em qualidade"
     - "Product Layer — move rapido, orientado a deadlines"
     - "Cada layer opera em sua velocidade natural"
     - "DS prioriza qualidade sobre velocidade; Product prioriza velocidade sobre perfeicao"
   quando_usar: "Quando ha tensao entre DS team e product teams sobre velocidade de entrega"
   Evidencia: "Product teams want the design system to hurry up! DS teams want product to slow down! Each layer needs to work at their respective pace."
   Source: data/brad-frost-research-compilation.md (Parte 3), data/brad-frost-quotes-index.md

10. [NOVO] Three-Tiered Token System
    steps:
      - "Global/Foundation tokens — colors, typography, spacing (reutilizavel broadly)"
      - "Component tokens — semantic naming (button-primary-bg vs blue-500)"
      - "Override tokens — raros, para casos muito especificos"
    quando_usar: "Ao arquitetar design tokens — evita explosao de 5000+ tokens component-specific"
    Evidencia: "Reserve component-specific tokens only for heavily-branded situations where variant wildly between themes." + caso do cliente com 5000+ tokens
    Source: data/brad-frost-research-compilation.md (Parte 4)

11. [NOVO] Real-Time UI Framework
    steps:
      - "Component visualization — componentes aparecem imediatamente quando referenciados na conversa"
      - "Product design realization — ideias abstratas viram prototipos tangiveis durante discussao"
      - "Infrastructure leverage — prototipos usam best practices da organizacao"
      - "Friction minimization — sem espera, sem handoffs, sem perda de contexto"
      - "Participatory design — reduz barreiras para nao-especialistas contribuirem"
    quando_usar: "Em reunioes de design/product — a reuniao se torna o prototipo"
    Evidencia: "If a picture is worth a thousand words, then a prototype is worth a thousand meetings. But what if the meeting IS the prototype?"
    Source: data/brad-frost-mius-structured.md (MIU_007, MIU_008)

12. [NOVO] Mouth Coding Pattern
    steps:
      - "Stakeholder fala ideia em linguagem natural"
      - "AI gera UI usando componentes do design system"
      - "DS constraints garantem qualidade e consistencia"
      - "Time esculpe junto em tempo real"
    quando_usar: "Colaboracao cross-disciplinar com membros nao-tecnicos — democratiza participacao no design"
    Evidencia: "Non-technical team members can 'mouth code' a product feature during a collaboration session and collectively we can see things come to life using the production-grade foundations of the design system."
    Source: data/brad-frost-mius-structured.md (MIU_002)

13. [NOVO] Component-Token Separation (Door Analogy)
    steps:
      - "Components = estrutura e funcionalidade (porta que abre e fecha)"
      - "Design tokens = estetica e customizacao (cor da pintura e macaneta)"
      - "Separar claramente os dois para escalar multi-brand/multi-product"
    quando_usar: "Ao decidir arquitetura de DS multi-marca — componentes reutilizaveis + tokens themaveis"
    Evidencia: "Door structure/functionality != paint color & hardware. Teams support wildly different aesthetic experiences without rebuilding components."
    Source: data/brad-frost-mius-structured.md (MIU_023)

14. [NOVO] Multi-All-The-Things Architecture
    steps:
      - "Identificar dimensoes de variacao (brands, products, platforms, modes, locales)"
      - "Componentes lidam com estrutura/funcionalidade (invariante)"
      - "Design tokens lidam com estetica/customizacao (variavel)"
      - "Naming convention que escala com todas dimensoes"
    quando_usar: "Organizacoes com multiplos produtos, marcas, plataformas simultaneos"
    Evidencia: Exemplos reais: Marriott (multi-brand hotels), Caterpillar Blocks, Apple, Verizon, Blend
    Source: data/brad-frost-mius-structured.md (MIU_024)

15. [NOVO] Global Design System Layer
    steps:
      - "HTML (primitivas: button, input)"
      - "Global Design System (componentes commodity: accordion, datepicker, tabs, modal)"
      - "Organization-specific DS (branded, customizado)"
      - "Product layer (recipes especificas de produto)"
    quando_usar: "Visao de futuro — camada canonical entre HTML e DS organizacionais para eliminar duplicacao global"
    Evidencia: "A Global Design System would centralize common UI components, reduce so much unnecessary duplication, integrate with any web-based tech stack."
    Source: data/brad-frost-mius-structured.md (MIU_031, MIU_032)

16. [NOVO] DS+AI vs Vibe Coding Distinction
    steps:
      - "DS+AI = AI deliberadamente constrained a componentes production-grade do DS"
      - "Vibe Coding = AI gera sem constraints (resultados inconsistentes)"
      - "DS fornece guardrails que previnem hallucination e inconsistencia"
    quando_usar: "Ao avaliar como AI gera UI — sempre exigir fundacao de DS"
    Evidencia: "This is what distinguishes DS+AI from vibe coding; the AI is deliberately constrained to using the high-quality design system materials."
    Source: data/brad-frost-mius-structured.md (MIU_003)

17. [NOVO] Coverage + Validation Pillars (para Agents)
    steps:
      - "Coverage: exemplos claros, estados documentados, constraints explicitos, composition patterns"
      - "Validation: testes automatizados, sign-off humano, verificacao de constraints, feedback loop"
    quando_usar: "Ao preparar DS para consumo por AI agents — dois pilares obrigatorios"
    Evidencia: "Coverage is about preparedness. Validation is about assurance. Both are necessary for agent-friendly design systems."
    Source: data/brad-frost-mius-structured.md (MIU_004)
